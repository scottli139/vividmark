# WYSIWYG 模式技术调研

> 调研目标：分析 Typora 的 WYSIWYG 实现方案，为 VividMark 设计合适的功能架构

## 1. Typora 实现方案分析

### 1.1 模式切换机制

从视频中观察到 Typora 采用**双模式设计**：

| 模式 | 快捷键 | 特点 |
|------|--------|------|
| **WYSIWYG 模式**（默认）| - | 所见即所得，直接编辑渲染后的内容 |
| **源代码模式** | `⌘/` | 显示原始 Markdown 源码，带行号 |

**关键发现**：
- Typora 默认就是 WYSIWYG 模式，**没有 Source/Preview/Split 的区分**
- 源代码模式只是一个「可切换的视图」，不是主要编辑方式
- 两种模式可以无缝切换，内容实时同步

### 1.2 WYSIWYG 核心交互

#### 编辑行为
1. **点击即编辑**：用户点击渲染后的内容任意位置，光标定位到该位置
2. **语法输入自动渲染**：
   - 输入 `## ABC` 按回车 → 自动渲染为二级标题
   - 输入 `### 呃呃呃` 按回车 → 自动渲染为三级标题
   - 表格、代码块等元素同样支持即时渲染
3. **渲染与编辑的无缝切换**：在渲染视图中直接编辑 Markdown 语法，失焦或回车后自动渲染

#### 技术实现推测
- 使用 `contenteditable` 或自定义编辑层
- 光标位置追踪（将渲染后的位置映射到 Markdown 源码位置）
- 语法解析器实时工作（可能是增量解析）
- 双向数据绑定（渲染视图 ↔ Markdown 源码）

### 1.3 界面元素

- **编辑区域**：单一编辑区，没有左右分栏
- **侧边栏**：文件树 + 大纲视图（与 VividMark 类似）
- **工具栏**：格式化按钮（加粗、斜体、标题等）

## 2. VividMark 现有模式分析

### 2.1 当前架构

| 模式 | 特点 | 适用场景 |
|------|------|---------|
| **Source** | 纯 Markdown 编辑 | 熟悉 Markdown 的用户，精确控制 |
| **Preview** | 只读预览 | 查看最终效果 |
| **Split** | 左源码右预览 | 学习 Markdown，边写边看 |

### 2.2 与 Typora 的对比

| 维度 | Typora | VividMark |
|------|--------|-----------|
| 默认体验 | WYSIWYG | Source/Split（需要选择）|
| 学习曲线 | 低（像 Word） | 中等（需要了解 Markdown）|
| 编辑效率 | 高（无需记忆语法） | 中高（需要切换模式）|
| 精确控制 | 中（隐藏语法细节） | 高（直接操作源码）|

## 3. 设计方案建议

### 方案 A：四模式共存（推荐）

保留现有三种模式，**新增 WYSIWYG 作为默认模式**：

```
[WYSIWYG] [Source] [Preview] [Split]
   ↑
默认选中
```

**优点**：
- 向后兼容，现有用户习惯不受影响
- 新用户默认获得类似 Typora 的体验
- 用户可以根据场景切换：
  - 快速编辑 → WYSIWYG
  - 精细调整 → Source
  - 查看效果 → Preview
  - 学习/对照 → Split

**缺点**：
- 模式较多，可能让用户困惑
- 需要维护四种模式的 UI 状态

### 方案 B：WYSIWYG 替换 Preview

将 Preview 改为可编辑的 WYSIWYG：

```
[Source] [WYSIWYG] [Split]
```

**优点**：
- 模式更简洁
- Preview 本身就有"查看最终效果"的含义，加入编辑顺理成章

**缺点**：
- 破坏向后兼容（Preview 变成可编辑）
- 用户可能想要一个真正的"只读预览"模式

### 方案 C：WYSIWYG + Source 双模式

类似 Typora 的设计：

```
[WYSIWYG] [Source]
```

**优点**：
- 最简洁
- 与 Typora 体验完全一致

**缺点**：
- 丢失 Split 模式的学习价值
- 现有用户可能需要适应

## 4. 技术实现建议

### 4.1 架构设计

```
┌─────────────────────────────────────────┐
│           WYSIWYG Editor                │
├─────────────────────────────────────────┤
│  ┌─────────────┐    ┌─────────────┐    │
│  │  Rendered   │ ↔  │  Markdown   │    │
│  │  Content    │    │   Source    │    │
│  │ (可编辑div)  │    │  (隐藏存储)  │    │
│  └─────────────┘    └─────────────┘    │
│         ↓                  ↑           │
│  ┌─────────────────────────────────┐   │
│  │      Position Mapping           │   │
│  │  (渲染位置 ↔ 源码位置映射)       │   │
│  └─────────────────────────────────┘   │
│         ↓                              │
│  ┌─────────────────────────────────┐   │
│  │    Incremental Parser           │   │
│  │   (增量解析，局部更新)           │   │
│  └─────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

### 4.2 关键技术点

#### 1. 可编辑渲染层
- 使用 `contenteditable` div 作为编辑容器
- 自定义渲染规则（复用现有 markdown-it 配置）
- 为每个可编辑元素添加 `data-source-range` 属性，记录对应源码位置

#### 2. 光标位置映射
```typescript
interface PositionMapping {
  // 渲染视图中的光标位置
  viewOffset: number
  // Markdown 源码中的对应位置
  sourceOffset: number
  // 元素级映射
  elementRange: [number, number]
}
```

#### 3. 增量解析
- 只解析用户当前编辑的块（block-level）
- 使用 markdown-it 的局部渲染能力
- 避免全文档重新渲染导致的闪烁

#### 4. 快捷键处理
- 复用现有的 `useKeyboardShortcuts`
- 在 WYSIWYG 模式下，快捷键直接操作渲染内容
- 例如：`Cmd+B` → 选中文字加粗 → 自动更新底层 Markdown

## 5. 与现有模式的共存策略

### 5.1 推荐方案：四模式共存 + 默认 WYSIWYG

**实现细节**：

1. **默认模式改为 WYSIWYG**
   ```typescript
   // editorStore.ts
   viewMode: 'wysiwyg' | 'source' | 'preview' | 'split' = 'wysiwyg'
   ```

2. **模式切换 UI**
   ```
   ┌─────────────────────────────────────────┐
   │  📝 WYSIWYG  │  📄 Source  │  👁 Preview  │  ↔️ Split  │
   └─────────────────────────────────────────┘
   ```

3. **首屏提示**
   - 首次使用 WYSIWYG 时，显示浮动提示：
   - "输入 `## 标题` 后按回车，自动渲染为二级标题"
   - 可关闭，不再显示

4. **快捷键统一**
   - `Cmd+/`：WYSIWYG ↔ Source 快速切换（类似 Typora）
   - `Cmd+Shift+P`：Preview 模式
   - `Cmd+\`：Split 模式

### 5.2 向后兼容

- 升级提示：向现有用户说明新增 WYSIWYG 模式
- 偏好设置：允许用户设置默认模式（如果用户习惯 Source，可设为默认）
